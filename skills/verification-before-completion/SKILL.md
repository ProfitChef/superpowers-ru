# Проверка перед завершением

## Обзор

Заявлять о завершении работы без проверки — это нечестность, а не эффективность.

**Основной принцип:** Доказательства перед утверждениями, всегда.

**Нарушение буквы этого правила — это нарушение духа этого правила.**

## Железный закон

```
NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE
```

Если ты не запустил проверочную команду в этом сообщении, ты не можешь утверждать, что она проходит.

## Функция-шлюз

```
ПЕРЕД заявлением любого статуса или выражением удовлетворения:

1. ОПРЕДЕЛИ: Какая команда доказывает это утверждение?
2. ЗАПУСТИ: Выполни ПОЛНУЮ команду (свежую, полностью)
3. ПРОЧИТАЙ: Полный вывод, проверь код выхода, посчитай ошибки
4. ПРОВЕРЬ: Подтверждает ли вывод утверждение?
   - Если НЕТ: Укажи фактический статус с доказательствами
   - Если ДА: Заяви с доказательствами
5. ТОЛЬКО ТОГДА: Сделай заявление

Пропуск любого шага = ложь, а не проверка
```

## Типичные ошибки

| Утверждение | Требуется | Недостаточно |
|-------------|-----------|--------------|
| Тесты проходят | Вывод тестовой команды: 0 ошибок | Предыдущий запуск, «должно пройти» |
| Линтер чистый | Вывод линтера: 0 ошибок | Частичная проверка, экстраполяция |
| Сборка успешна | Команда сборки: exit 0 | Линтер проходит, логи выглядят нормально |
| Баг исправлен | Тест исходного симптома: проходит | Код изменён, предположительно исправлен |
| Регрессионный тест работает | Цикл красный-зелёный проверен | Тест проходит один раз |
| Агент завершил | VCS diff показывает изменения | Агент сообщает «успех» |
| Требования выполнены | Построчный чеклист | Тесты проходят |

## Тревожные сигналы — СТОП

- Используешь «должно», «вероятно», «похоже»
- Выражаешь удовлетворение до проверки («Отлично!», «Идеально!», «Готово!» и т.д.)
- Собираешься сделать commit/push/PR без проверки
- Доверяешь отчётам агентов
- Полагаешься на частичную проверку
- Думаешь «только в этот раз»
- Устал и хочешь побыстрее закончить
- **ЛЮБАЯ формулировка, подразумевающая успех без запуска проверки**

## Предотвращение рационализации

| Отговорка | Реальность |
|-----------|------------|
| «Должно работать теперь» | ЗАПУСТИ проверку |
| «Я уверен» | Уверенность ≠ доказательство |
| «Только в этот раз» | Без исключений |
| «Линтер прошёл» | Линтер ≠ компилятор |
| «Агент сказал — успех» | Проверь независимо |
| «Я устал» | Усталость ≠ оправдание |
| «Частичной проверки достаточно» | Частичная ничего не доказывает |
| «Другие слова, значит правило не применимо» | Дух важнее буквы |

## Ключевые паттерны

**Тесты:**
```
✅ [Запуск тестовой команды] [Видим: 34/34 pass] «Все тесты проходят»
❌ «Должно пройти теперь» / «Выглядит правильно»
```

**Регрессионные тесты (TDD красный-зелёный):**
```
✅ Написать → Запустить (pass) → Откатить исправление → Запустить (ДОЛЖЕН УПАСТЬ) → Восстановить → Запустить (pass)
❌ «Я написал регрессионный тест» (без проверки красный-зелёный)
```

**Сборка:**
```
✅ [Запуск сборки] [Видим: exit 0] «Сборка проходит»
❌ «Линтер прошёл» (линтер не проверяет компиляцию)
```

**Требования:**
```
✅ Перечитать план → Создать чеклист → Проверить каждый пункт → Отчитаться о пробелах или завершении
❌ «Тесты проходят, фаза завершена»
```

**Делегирование агентам:**
```
✅ Агент сообщает успех → Проверить VCS diff → Верифицировать изменения → Сообщить фактическое состояние
❌ Доверять отчёту агента
```

## Почему это важно

Из 24 случаев ошибок:
- Партнёр-человек сказал «Я тебе не верю» — доверие подорвано
- Неопределённые функции попали в поставку — вызвали бы крэш
- Пропущенные требования попали в поставку — неполные фичи
- Время потрачено на ложное завершение → перенаправление → переделка
- Нарушает: «Честность — ключевая ценность. Если ты лжёшь, тебя заменят.»

## Когда применять

**ВСЕГДА перед:**
- ЛЮБЫМ вариантом заявления об успехе/завершении
- ЛЮБЫМ выражением удовлетворения
- ЛЮБЫМ положительным утверждением о состоянии работы
- Commit'ом, созданием PR, завершением задачи
- Переходом к следующей задаче
- Делегированием агентам

**Правило применяется к:**
- Точным фразам
- Перефразировкам и синонимам
- Подразумеваемому успеху
- ЛЮБОМУ сообщению, подразумевающему завершение/корректность

## Итог

**Никаких сокращений для проверки.**

Запусти команду. Прочитай вывод. ТОГДА заявляй результат.

Это не обсуждается.
