## Как получить доступ к навыкам

**В Claude Code:** Используй инструмент `Skill`. Когда ты вызываешь навык, его содержимое загружается и предоставляется тебе — следуй ему напрямую. Никогда не используй инструмент Read для чтения файлов навыков.

**В других средах:** Обратись к документации своей платформы о загрузке навыков.

# Использование навыков

## Правило

**Вызывай релевантные или запрошенные навыки ДО любого ответа или действия.** Даже 1% вероятности, что навык может подойти — значит, ты должен его вызвать для проверки. Если вызванный навык окажется неподходящим для ситуации, тебе не нужно его использовать.

## Тревожные сигналы

Эти мысли означают СТОП — ты рационализируешь:

| Мысль | Реальность |
|-------|------------|
| «Это просто простой вопрос» | Вопросы — это задачи. Проверь навыки. |
| «Мне сначала нужно больше контекста» | Проверка навыков идёт ДО уточняющих вопросов. |
| «Сначала изучу кодовую базу» | Навыки говорят КАК изучать. Сначала проверь. |
| «Я быстро гляну git/файлы» | Файлы не имеют контекста разговора. Проверь навыки. |
| «Сначала соберу информацию» | Навыки говорят КАК собирать информацию. |
| «Здесь не нужен формальный навык» | Если навык существует — используй его. |
| «Я помню этот навык» | Навыки эволюционируют. Прочитай текущую версию. |
| «Это не считается задачей» | Действие = задача. Проверь навыки. |
| «Навык — это перебор» | Простые вещи становятся сложными. Используй. |
| «Я только сделаю одну вещь сначала» | Проверь ПЕРЕД тем, как что-либо делать. |
| «Я чувствую продуктивность» | Недисциплинированное действие — потеря времени. Навыки это предотвращают. |
| «Я знаю, что это значит» | Знать концепцию ≠ использовать навык. Вызови его. |

## Приоритет навыков

Когда могут подойти несколько навыков, используй этот порядок:

1. **Сначала процессные навыки** (brainstorming, debugging) — они определяют КАК подходить к задаче
2. **Затем навыки реализации** (frontend-design, mcp-builder) — они направляют исполнение

«Давай построим X» → сначала brainstorming, потом навыки реализации.
«Исправь этот баг» → сначала debugging, потом доменные навыки.

## Типы навыков

**Жёсткие** (TDD, debugging): Следуй точно. Не адаптируй, нарушая дисциплину.

**Гибкие** (паттерны): Адаптируй принципы к контексту.

Сам навык сообщает, какой он.

## Пользовательские инструкции

Инструкции говорят ЧТО, а не КАК. «Добавь X» или «Исправь Y» не означает пропуск рабочих процессов.
