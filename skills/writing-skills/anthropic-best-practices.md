# Лучшие практики создания навыков

> Узнайте, как писать эффективные навыки (Skill), которые Claude сможет находить и успешно использовать.

Хорошие навыки лаконичны, хорошо структурированы и проверены на реальном использовании. Это руководство содержит практические рекомендации, которые помогут вам писать навыки, которые Claude сможет находить и эффективно применять.

Концептуальную информацию о том, как работают навыки, см. в [обзоре навыков](/en/docs/agents-and-tools/agent-skills/overview).

## Основные принципы

### Краткость — это ключ

[Контекстное окно](https://platform.claude.com/docs/en/build-with-claude/context-windows) — это общий ресурс. Ваш навык делит контекстное окно со всем остальным, что нужно Claude, включая:

* Системный промпт
* Историю разговора
* Метаданные других навыков
* Ваш собственно запрос

Не каждый токен в навыке имеет немедленную стоимость. При запуске предзагружаются только метаданные (имя и описание) всех навыков. Claude читает SKILL.md только когда навык становится релевантным, а дополнительные файлы — только по необходимости. Тем не менее, краткость SKILL.md важна: как только Claude его загрузит, каждый токен конкурирует с историей разговора и другим контекстом.

**Исходное допущение**: Claude уже очень умён

Добавляйте только тот контекст, которого у Claude ещё нет. Проверяйте каждый фрагмент информации:

* «Действительно ли Claude нужно это объяснение?»
* «Могу ли я предположить, что Claude это знает?»
* «Оправдывает ли этот абзац свою стоимость в токенах?»

**Хороший пример: лаконично** (примерно 50 токенов):

````markdown  theme={null}
## Извлечение текста из PDF

Используйте pdfplumber для извлечения текста:

```python
import pdfplumber

with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```
````

**Плохой пример: слишком многословно** (примерно 150 токенов):

```markdown  theme={null}
## Извлечение текста из PDF

PDF (Portable Document Format) — это распространённый формат файлов, который
содержит текст, изображения и другой контент. Чтобы извлечь текст из PDF,
вам понадобится библиотека. Существует множество библиотек для обработки PDF,
но мы рекомендуем pdfplumber, потому что она проста в использовании и
подходит для большинства случаев. Сначала установите её через pip.
Затем используйте код ниже...
```

Лаконичная версия предполагает, что Claude знает, что такое PDF и как работают библиотеки.

### Устанавливайте подходящую степень свободы

Уровень детализации должен соответствовать хрупкости и вариативности задачи.

**Высокая свобода** (текстовые инструкции):

Используйте, когда:

* Допустимы несколько подходов
* Решения зависят от контекста
* Эвристики направляют подход

Пример:

```markdown  theme={null}
## Процесс код-ревью

1. Проанализировать структуру и организацию кода
2. Проверить на потенциальные баги и граничные случаи
3. Предложить улучшения читаемости и поддерживаемости
4. Убедиться в соответствии проектным соглашениям
```

**Средняя свобода** (псевдокод или скрипты с параметрами):

Используйте, когда:

* Существует предпочтительный паттерн
* Некоторые вариации допустимы
* Конфигурация влияет на поведение

Пример:

````markdown  theme={null}
## Генерация отчёта

Используйте этот шаблон и адаптируйте по необходимости:

```python
def generate_report(data, format="markdown", include_charts=True):
    # Обработать данные
    # Сгенерировать вывод в указанном формате
    # Опционально включить визуализации
```
````

**Низкая свобода** (конкретные скрипты, мало или без параметров):

Используйте, когда:

* Операции хрупкие и подвержены ошибкам
* Единообразие критически важно
* Необходимо следовать определённой последовательности

Пример:

````markdown  theme={null}
## Миграция базы данных

Выполните именно этот скрипт:

```bash
python scripts/migrate.py --verify --backup
```

Не изменяйте команду и не добавляйте дополнительные флаги.
````

**Аналогия**: Представьте Claude как робота, исследующего путь:

* **Узкий мост с обрывами по бокам**: Есть только один безопасный путь вперёд. Обеспечьте конкретные ограждения и точные инструкции (низкая свобода). Пример: миграции БД, которые должны выполняться в точной последовательности.
* **Открытое поле без препятствий**: Множество путей ведут к успеху. Задайте общее направление и доверьтесь Claude найти лучший маршрут (высокая свобода). Пример: код-ревью, где контекст определяет лучший подход.

### Тестируйте на всех моделях, которые планируете использовать

Навыки действуют как дополнения к моделям, поэтому эффективность зависит от базовой модели. Тестируйте навык со всеми моделями, с которыми планируете его использовать.

**Что учитывать при тестировании на разных моделях**:

* **Claude Haiku** (быстрая, экономичная): Достаточно ли руководства даёт навык?
* **Claude Sonnet** (сбалансированная): Навык понятный и эффективный?
* **Claude Opus** (мощная аналитика): Навык не перегружен лишними пояснениями?

То, что отлично работает для Opus, может потребовать больше деталей для Haiku. Если вы планируете использовать навык с несколькими моделями, стремитесь к инструкциям, которые хорошо работают со всеми.

## Структура навыка

<Note>
  **YAML-метаданные**: Метаданные SKILL.md поддерживают два поля:

  * `name` — человекочитаемое название навыка (максимум 64 символа)
  * `description` — однострочное описание того, что делает навык и когда его использовать (максимум 1024 символа)

  Подробности о полной структуре навыка см. в [обзоре навыков](/en/docs/agents-and-tools/agent-skills/overview#skill-structure).
</Note>

### Соглашения об именовании

Используйте единообразные паттерны именования, чтобы на навыки было проще ссылаться и обсуждать их. Мы рекомендуем использовать **форму герундия** (глагол + -ing) для названий навыков, так как она чётко описывает деятельность или возможность, которую предоставляет навык.

**Хорошие примеры именования (форма герундия)**:

* "Processing PDFs"
* "Analyzing spreadsheets"
* "Managing databases"
* "Testing code"
* "Writing documentation"

**Допустимые альтернативы**:

* Именные группы: "PDF Processing", "Spreadsheet Analysis"
* Ориентированные на действие: "Process PDFs", "Analyze Spreadsheets"

**Избегайте**:

* Размытых названий: "Helper", "Utils", "Tools"
* Слишком общих: "Documents", "Data", "Files"
* Непоследовательных паттернов внутри вашей коллекции навыков

Единообразное именование упрощает:

* Ссылки на навыки в документации и разговорах
* Понимание того, что делает навык, с первого взгляда
* Организацию и поиск среди множества навыков
* Поддержание профессиональной, цельной библиотеки навыков

### Написание эффективных описаний

Поле `description` обеспечивает обнаружение навыка и должно включать как то, что навык делает, так и когда его использовать.

<Warning>
  **Всегда пишите от третьего лица**. Описание внедряется в системный промпт, и непоследовательная точка зрения может вызвать проблемы с обнаружением.

  * **Хорошо:** "Processes Excel files and generates reports"
  * **Избегайте:** "I can help you process Excel files"
  * **Избегайте:** "You can use this to process Excel files"
</Warning>

**Будьте конкретны и включайте ключевые термины**. Включайте как то, что навык делает, так и конкретные триггеры/контексты, когда его использовать.

У каждого навыка ровно одно поле описания. Описание критически важно для выбора навыка: Claude использует его, чтобы выбрать нужный навык из потенциально 100+ доступных. Ваше описание должно давать достаточно деталей, чтобы Claude знал, когда выбрать этот навык, а остальная часть SKILL.md предоставляет детали реализации.

Эффективные примеры:

**Навык обработки PDF:**

```yaml  theme={null}
description: Extract text and tables from PDF files, fill forms, merge documents. Use when working with PDF files or when the user mentions PDFs, forms, or document extraction.
```

**Навык анализа Excel:**

```yaml  theme={null}
description: Analyze Excel spreadsheets, create pivot tables, generate charts. Use when analyzing Excel files, spreadsheets, tabular data, or .xlsx files.
```

**Навык помощника для commit-сообщений:**

```yaml  theme={null}
description: Generate descriptive commit messages by analyzing git diffs. Use when the user asks for help writing commit messages or reviewing staged changes.
```

Избегайте размытых описаний вроде этих:

```yaml  theme={null}
description: Helps with documents
```

```yaml  theme={null}
description: Processes data
```

```yaml  theme={null}
description: Does stuff with files
```

### Паттерны постепенного раскрытия

SKILL.md служит обзором, который направляет Claude к детальным материалам по мере необходимости, как оглавление в руководстве для новичка. Объяснение того, как работает постепенное раскрытие, см. в разделе [Как работают навыки](/en/docs/agents-and-tools/agent-skills/overview#how-skills-work) обзора.

**Практические рекомендации:**

* Держите тело SKILL.md до 500 строк для оптимальной производительности
* Разделяйте контент на отдельные файлы при приближении к этому лимиту
* Используйте паттерны ниже для эффективной организации инструкций, кода и ресурсов

Полная структура каталога навыка может выглядеть так:

```
pdf/
├── SKILL.md              # Основные инструкции (загружаются при активации)
├── FORMS.md              # Руководство по заполнению форм (загружается по необходимости)
├── reference.md          # Справочник API (загружается по необходимости)
├── examples.md           # Примеры использования (загружаются по необходимости)
└── scripts/
    ├── analyze_form.py   # Утилитный скрипт (выполняется, не загружается)
    ├── fill_form.py      # Скрипт заполнения форм
    └── validate.py       # Скрипт валидации
```

#### Паттерн 1: Высокоуровневое руководство со ссылками

````markdown  theme={null}
---
name: PDF Processing
description: Extracts text and tables from PDF files, fills forms, and merges documents. Use when working with PDF files or when the user mentions PDFs, forms, or document extraction.
---

# Обработка PDF

## Быстрый старт

Извлечение текста с помощью pdfplumber:
```python
import pdfplumber
with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```

## Продвинутые возможности

**Заполнение форм**: См. [FORMS.md](FORMS.md) — полное руководство
**Справочник API**: См. [REFERENCE.md](REFERENCE.md) — все методы
**Примеры**: См. [EXAMPLES.md](EXAMPLES.md) — типовые паттерны
````

Claude загружает FORMS.md, REFERENCE.md или EXAMPLES.md только по необходимости.

#### Паттерн 2: Организация по предметным областям

Для навыков с несколькими предметными областями организуйте контент по областям, чтобы не загружать нерелевантный контекст. Когда пользователь спрашивает о метриках продаж, Claude нужно прочитать только схемы, связанные с продажами, а не финансовые или маркетинговые данные. Это экономит токены и фокусирует контекст.

```
bigquery-skill/
├── SKILL.md (обзор и навигация)
└── reference/
    ├── finance.md (выручка, метрики биллинга)
    ├── sales.md (сделки, воронка продаж)
    ├── product.md (использование API, фичи)
    └── marketing.md (кампании, атрибуция)
```

````markdown SKILL.md theme={null}
# Анализ данных BigQuery

## Доступные наборы данных

**Финансы**: Выручка, ARR, биллинг → См. [reference/finance.md](reference/finance.md)
**Продажи**: Сделки, воронка, аккаунты → См. [reference/sales.md](reference/sales.md)
**Продукт**: Использование API, фичи, внедрение → См. [reference/product.md](reference/product.md)
**Маркетинг**: Кампании, атрибуция, email → См. [reference/marketing.md](reference/marketing.md)

## Быстрый поиск

Поиск конкретных метрик с помощью grep:

```bash
grep -i "revenue" reference/finance.md
grep -i "pipeline" reference/sales.md
grep -i "api usage" reference/product.md
```
````

### Избегайте глубоко вложенных ссылок

Claude может частично читать файлы, на которые ссылаются из других файлов-ссылок. При встрече с вложенными ссылками Claude может использовать команды вроде `head -100` для предпросмотра содержимого вместо чтения файлов целиком, что приводит к неполной информации.

**Держите ссылки на одном уровне вложенности от SKILL.md**. Все справочные файлы должны быть связаны напрямую из SKILL.md, чтобы Claude читал полные файлы при необходимости.

**Плохой пример: слишком глубоко**:

```markdown  theme={null}
# SKILL.md
См. [advanced.md](advanced.md)...

# advanced.md
См. [details.md](details.md)...

# details.md
Вот собственно информация...
```

**Хороший пример: один уровень вложенности**:

```markdown  theme={null}
# SKILL.md

**Базовое использование**: [инструкции в SKILL.md]
**Продвинутые возможности**: См. [advanced.md](advanced.md)
**Справочник API**: См. [reference.md](reference.md)
**Примеры**: См. [examples.md](examples.md)
```

### Структурируйте длинные справочные файлы с оглавлением

Для справочных файлов длиннее 100 строк включите оглавление в начало. Это гарантирует, что Claude сможет увидеть полный объём доступной информации даже при частичном чтении.

**Пример**:

```markdown  theme={null}
# Справочник API

## Содержание
- Аутентификация и настройка
- Основные методы (создание, чтение, обновление, удаление)
- Продвинутые возможности (пакетные операции, вебхуки)
- Паттерны обработки ошибок
- Примеры кода

## Аутентификация и настройка
...

## Основные методы
...
```

Claude может затем прочитать файл целиком или перейти к нужным разделам.

Подробности о том, как эта файловая архитектура обеспечивает постепенное раскрытие, см. в разделе [Среда выполнения](#runtime-environment) ниже в разделе «Продвинутое».

## Рабочие процессы и циклы обратной связи

### Используйте рабочие процессы для сложных задач

Разбивайте сложные операции на чёткие последовательные шаги. Для особенно сложных процессов предоставьте чеклист, который Claude может скопировать в свой ответ и отмечать по мере выполнения.

**Пример 1: Процесс синтеза исследований** (для навыков без кода):

````markdown  theme={null}
## Процесс синтеза исследований

Скопируйте этот чеклист и отслеживайте прогресс:

```
Прогресс исследования:
- [ ] Шаг 1: Прочитать все исходные документы
- [ ] Шаг 2: Выявить ключевые темы
- [ ] Шаг 3: Перекрёстная проверка утверждений
- [ ] Шаг 4: Создать структурированное резюме
- [ ] Шаг 5: Проверить цитирование
```

**Шаг 1: Прочитать все исходные документы**

Просмотрите каждый документ в каталоге `sources/`. Запишите основные аргументы и подтверждающие доказательства.

**Шаг 2: Выявить ключевые темы**

Ищите паттерны по всем источникам. Какие темы повторяются? Где источники совпадают или расходятся?

**Шаг 3: Перекрёстная проверка утверждений**

Для каждого крупного утверждения убедитесь, что оно присутствует в исходном материале. Отметьте, какой источник подтверждает каждый тезис.

**Шаг 4: Создать структурированное резюме**

Организуйте выводы по темам. Включите:
- Основное утверждение
- Подтверждающие доказательства из источников
- Противоречащие точки зрения (если есть)

**Шаг 5: Проверить цитирование**

Убедитесь, что каждое утверждение ссылается на правильный исходный документ. Если цитирование неполное, вернитесь к Шагу 3.
````

Этот пример показывает, как рабочие процессы применяются к аналитическим задачам, не требующим кода. Паттерн чеклиста работает для любого сложного многошагового процесса.

**Пример 2: Процесс заполнения PDF-форм** (для навыков с кодом):

````markdown  theme={null}
## Процесс заполнения PDF-форм

Скопируйте этот чеклист и отмечайте пункты по мере выполнения:

```
Прогресс задачи:
- [ ] Шаг 1: Анализ формы (запустить analyze_form.py)
- [ ] Шаг 2: Создание маппинга полей (редактировать fields.json)
- [ ] Шаг 3: Валидация маппинга (запустить validate_fields.py)
- [ ] Шаг 4: Заполнение формы (запустить fill_form.py)
- [ ] Шаг 5: Проверка результата (запустить verify_output.py)
```

**Шаг 1: Анализ формы**

Запустите: `python scripts/analyze_form.py input.pdf`

Это извлечёт поля формы и их расположение, сохранив в `fields.json`.

**Шаг 2: Создание маппинга полей**

Отредактируйте `fields.json`, добавив значения для каждого поля.

**Шаг 3: Валидация маппинга**

Запустите: `python scripts/validate_fields.py fields.json`

Исправьте все ошибки валидации перед продолжением.

**Шаг 4: Заполнение формы**

Запустите: `python scripts/fill_form.py input.pdf fields.json output.pdf`

**Шаг 5: Проверка результата**

Запустите: `python scripts/verify_output.py output.pdf`

Если проверка не пройдена, вернитесь к Шагу 2.
````

Чёткие шаги не дают Claude пропустить критичную валидацию. Чеклист помогает и Claude, и вам отслеживать прогресс в многошаговых процессах.

### Внедряйте циклы обратной связи

**Типовой паттерн**: Запустить валидатор → исправить ошибки → повторить

Этот паттерн значительно повышает качество результата.

**Пример 1: Соответствие стилевому руководству** (для навыков без кода):

```markdown  theme={null}
## Процесс ревью контента

1. Составьте контент согласно рекомендациям в STYLE_GUIDE.md
2. Проверьте по чеклисту:
   - Проверка единообразия терминологии
   - Примеры соответствуют стандартному формату
   - Все обязательные разделы присутствуют
3. Если найдены проблемы:
   - Отметьте каждую проблему со ссылкой на конкретный раздел
   - Исправьте контент
   - Снова проверьте по чеклисту
4. Продолжайте только когда все требования выполнены
5. Финализируйте и сохраните документ
```

Здесь показан паттерн цикла валидации с использованием справочных документов вместо скриптов. «Валидатором» выступает STYLE\_GUIDE.md, а Claude выполняет проверку, читая и сравнивая.

**Пример 2: Процесс редактирования документа** (для навыков с кодом):

```markdown  theme={null}

1. Внесите правки в `word/document.xml`
2. **Немедленно валидируйте**: `python ooxml/scripts/validate.py unpacked_dir/`

   - Запустите валидацию снова
4. **Продолжайте только когда валидация пройдена**
5. Пересоберите: `python ooxml/scripts/pack.py unpacked_dir/ output.docx`
6. Протестируйте результирующий документ
```

Цикл валидации позволяет ловить ошибки на ранних этапах.

## Оценка и итерация

### Сначала создайте оценки

**Создавайте оценки ДО написания обширной документации.** Это гарантирует, что ваш навык решает реальные проблемы, а не документирует воображаемые.

**Разработка через оценки:**

1. **Выявите пробелы**: Запустите Claude на типичных задачах без навыка. Задокументируйте конкретные неудачи или недостающий контекст
2. **Создайте оценки**: Подготовьте три сценария, тестирующие эти пробелы
3. **Установите базовую линию**: Измерьте производительность Claude без навыка
4. **Напишите минимальные инструкции**: Создайте ровно столько контента, чтобы закрыть пробелы и пройти оценки
5. **Итерируйте**: Запускайте оценки, сравнивайте с базовой линией и улучшайте

Этот подход гарантирует, что вы решаете реальные проблемы, а не предвосхищаете требования, которые могут никогда не материализоваться.

**Структура оценки**:

```json  theme={null}
{
  "skills": ["pdf-processing"],
  "query": "Extract all text from this PDF file and save it to output.txt",
  "files": ["test-files/document.pdf"],
  "expected_behavior": [
    "Successfully reads the PDF file using an appropriate PDF processing library or command-line tool",
    "Extracts text content from all pages in the document without missing any pages",
    "Saves the extracted text to a file named output.txt in a clear, readable format"
  ]
}
```

<Note>
  Этот пример демонстрирует оценку на основе данных с простой тестовой рубрикой. В настоящее время мы не предоставляем встроенного способа запуска этих оценок. Пользователи могут создать собственную систему оценки. Оценки — это ваш источник истины для измерения эффективности навыка.
</Note>

### Разрабатывайте навыки итеративно с помощью Claude

Наиболее эффективный процесс разработки навыков включает самого Claude. Работайте с одним экземпляром Claude («Claude A»), чтобы создать навык, который будет использоваться другими экземплярами («Claude B»). Claude A помогает вам проектировать и улучшать инструкции, а Claude B тестирует их на реальных задачах. Это работает, потому что модели Claude понимают как написание эффективных агентных инструкций, так и то, какая информация нужна агентам.

**Создание нового навыка:**

1. **Выполните задачу без навыка**: Проработайте проблему с Claude A через обычный промптинг. По ходу работы вы естественным образом будете давать контекст, объяснять предпочтения и делиться процедурными знаниями. Обратите внимание, какую информацию вы предоставляете повторно.

2. **Определите повторяемый паттерн**: После выполнения задачи определите, какой контекст вы предоставили, который будет полезен для аналогичных будущих задач.

   **Пример**: Если вы прорабатывали анализ BigQuery, вы могли предоставить имена таблиц, определения полей, правила фильтрации (например, «всегда исключать тестовые аккаунты») и типовые паттерны запросов.

3. **Попросите Claude A создать навык**: «Создай навык, который закрепит паттерн анализа BigQuery, который мы только что использовали. Включи схемы таблиц, соглашения об именовании и правило о фильтрации тестовых аккаунтов.»

   <Tip>
     Модели Claude нативно понимают формат и структуру навыков. Вам не нужны специальные системные промпты или «навык написания навыков», чтобы Claude помог создать навык. Просто попросите Claude создать навык, и он сгенерирует правильно структурированный контент SKILL.md с соответствующими метаданными и телом.
   </Tip>

4. **Проверьте на лаконичность**: Убедитесь, что Claude A не добавил лишних пояснений. Спросите: «Убери объяснение того, что такое win rate — Claude это уже знает.»

5. **Улучшите информационную архитектуру**: Попросите Claude A организовать контент более эффективно. Например: «Организуй это так, чтобы схема таблиц была в отдельном справочном файле. Позже мы можем добавить больше таблиц.»

6. **Протестируйте на похожих задачах**: Используйте навык с Claude B (свежий экземпляр с загруженным навыком) на смежных кейсах. Посмотрите, находит ли Claude B нужную информацию, правильно ли применяет правила и успешно ли справляется с задачей.

7. **Итерируйте на основе наблюдений**: Если Claude B затрудняется или что-то упускает, вернитесь к Claude A с конкретикой: «Когда Claude использовал этот навык, он забыл отфильтровать по дате для Q4. Стоит ли добавить раздел о паттернах фильтрации по датам?»

**Итерация существующих навыков:**

Тот же иерархический паттерн продолжается при улучшении навыков. Вы чередуете:

* **Работу с Claude A** (эксперт, помогающий улучшать навык)
* **Тестирование с Claude B** (агент, использующий навык для реальной работы)
* **Наблюдение за поведением Claude B** и возврат инсайтов Claude A

1. **Используйте навык в реальных процессах**: Давайте Claude B (с загруженным навыком) реальные задачи, а не тестовые сценарии

2. **Наблюдайте за поведением Claude B**: Отмечайте, где он затрудняется, справляется или делает неожиданный выбор

   **Пример наблюдения**: «Когда я попросил Claude B создать региональный отчёт о продажах, он написал запрос, но забыл отфильтровать тестовые аккаунты, хотя навык упоминает это правило.»

3. **Вернитесь к Claude A для улучшений**: Поделитесь текущим SKILL.md и опишите, что наблюдали. Спросите: «Я заметил, что Claude B забыл отфильтровать тестовые аккаунты в региональном отчёте. Навык упоминает фильтрацию, но может, она недостаточно выделена?»

4. **Оцените предложения Claude A**: Claude A может предложить реорганизацию, чтобы правила были заметнее, использование более сильных формулировок вроде «ОБЯЗАТЕЛЬНО фильтровать» вместо «всегда фильтровать» или реструктуризацию раздела с процессом.

5. **Примените и протестируйте изменения**: Обновите навык улучшениями от Claude A, затем снова протестируйте с Claude B на похожих запросах

6. **Повторяйте по мере использования**: Продолжайте этот цикл «наблюдение—улучшение—тест» по мере появления новых сценариев. Каждая итерация улучшает навык на основе реального поведения агента, а не допущений.

**Сбор обратной связи от команды:**

1. Поделитесь навыками с коллегами и понаблюдайте за их использованием
2. Спросите: Навык активируется, когда ожидается? Инструкции понятны? Чего не хватает?
3. Учтите обратную связь, чтобы закрыть слепые зоны ваших собственных паттернов использования

**Почему этот подход работает**: Claude A понимает потребности агента, вы обеспечиваете экспертизу предметной области, Claude B выявляет пробелы через реальное использование, а итеративное улучшение совершенствует навыки на основе наблюдаемого поведения, а не допущений.

### Наблюдайте, как Claude навигирует по навыкам

Итерируя навыки, обращайте внимание на то, как Claude фактически использует их на практике. Следите за:

* **Неожиданными путями исследования**: Claude читает файлы в порядке, который вы не предполагали? Это может означать, что ваша структура не так интуитивна, как вы думали
* **Пропущенными связями**: Claude не следует ссылкам на важные файлы? Ваши ссылки могут быть недостаточно явными или заметными
* **Чрезмерной зависимостью от отдельных разделов**: Если Claude постоянно перечитывает один и тот же файл, возможно, этот контент стоит перенести в основной SKILL.md
* **Игнорируемым контентом**: Если Claude никогда не обращается к прикреплённому файлу, он может быть ненужным или плохо анонсированным в основных инструкциях

Итерируйте на основе этих наблюдений, а не допущений. Поля `name` и `description` в метаданных навыка особенно критичны. Claude использует их при решении, активировать ли навык в ответ на текущую задачу. Убедитесь, что они чётко описывают, что делает навык и когда его следует использовать.

## Антипаттерны

### Избегайте путей в стиле Windows

Всегда используйте прямые слэши в путях к файлам, даже в Windows:

* ✓ **Хорошо**: `scripts/helper.py`, `reference/guide.md`
* ✗ **Избегайте**: `scripts\helper.py`, `reference\guide.md`

Unix-стиль путей работает на всех платформах, тогда как Windows-стиль вызывает ошибки в Unix-системах.

### Избегайте избытка вариантов

Не предлагайте несколько подходов, если в этом нет необходимости:

````markdown  theme={null}
**Плохой пример: слишком много вариантов** (запутывает):
"Можно использовать pypdf, или pdfplumber, или PyMuPDF, или pdf2image, или..."

**Хороший пример: дайте вариант по умолчанию** (с запасным выходом):

import pdfplumber
````

## Продвинутое: навыки с исполняемым кодом

Разделы ниже посвящены навыкам, включающим исполняемые скрипты. Если ваш навык использует только инструкции в markdown, переходите к [Чеклист эффективных навыков](#checklist-for-effective-skills).

### Решайте, а не перекладывайте

При написании скриптов для навыков обрабатывайте ошибки сами, а не перекладывайте на Claude.

**Хороший пример: явная обработка ошибок**:

```python  theme={null}
def process_file(path):
    """Обработать файл, создав его при отсутствии."""
    try:
        with open(path) as f:
            return f.read()
    except FileNotFoundError:
        # Создать файл с контентом по умолчанию вместо падения
        print(f"File {path} not found, creating default")
        with open(path, 'w') as f:
            f.write('')
        return ''
    except PermissionError:
        # Предоставить альтернативу вместо падения
        print(f"Cannot access {path}, using default")
        return ''
```

**Плохой пример: переложить на Claude**:

```python  theme={null}
def process_file(path):
    # Просто упасть и пусть Claude разбирается
    return open(path).read()
```

Конфигурационные параметры также должны быть обоснованы и задокументированы, чтобы избежать «магических констант» (закон Оустерхаута). Если вы не знаете правильное значение, как Claude его определит?

**Хороший пример: самодокументирующийся**:

```python  theme={null}
# HTTP-запросы обычно завершаются в течение 30 секунд
# Увеличенный таймаут учитывает медленные соединения
REQUEST_TIMEOUT = 30

# Три повтора балансируют надёжность и скорость
# Большинство прерывистых сбоев устраняются ко второму повтору
MAX_RETRIES = 3
```

**Плохой пример: магические числа**:

```python  theme={null}
TIMEOUT = 47  # Почему 47?
RETRIES = 5   # Почему 5?
```

**Преимущества утилитных скриптов**:

* Более надёжны, чем сгенерированный код
* Экономят токены (не нужно включать код в контекст)
* Экономят время (не требуется генерация кода)
* Обеспечивают единообразие использования

* **Прочитать как справку** (для сложной логики): «См. `analyze_form.py` для алгоритма извлечения полей»

Для большинства утилитных скриптов предпочтительно выполнение, так как оно более надёжно и эффективно. Подробности см. в разделе [Среда выполнения](#runtime-environment) ниже.

````markdown  theme={null}
## Утилитные скрипты

**analyze_form.py**: Извлечение всех полей формы из PDF

```

Формат вывода:

}

python scripts/validate_boxes.py fields.json
````

Когда входные данные можно отрендерить как изображения, попросите Claude проанализировать их:

````markdown  theme={null}
## Анализ макета формы

1. Конвертируйте PDF в изображения:
   ```bash
   python scripts/pdf_to_images.py form.pdf
   ```

2. Проанализируйте каждое изображение страницы для определения полей формы
3. Claude может визуально видеть расположение и типы полей
````

<Note>
  В этом примере вам нужно написать скрипт `pdf_to_images.py`.
</Note>

Визуальные возможности Claude помогают понять макеты и структуры.

### Создавайте верифицируемые промежуточные результаты

Когда Claude выполняет сложные, открытые задачи, он может ошибаться. Паттерн «планируй—валидируй—выполняй» ловит ошибки на ранней стадии: Claude сначала создаёт план в структурированном формате, затем валидирует его скриптом перед выполнением.

**Пример**: Представьте, что вы просите Claude обновить 50 полей формы в PDF на основе таблицы. Без валидации Claude может обратиться к несуществующим полям, создать конфликтующие значения, пропустить обязательные поля или некорректно применить обновления.

**Решение**: Используйте паттерн рабочего процесса, показанный выше (заполнение PDF-формы), но добавьте промежуточный файл `changes.json`, который валидируется перед применением изменений. Процесс становится: анализ → **создание файла плана** → **валидация плана** → выполнение → проверка.

**Почему этот паттерн работает:**

* **Ловит ошибки рано**: Валидация находит проблемы до применения изменений
* **Машинно-верифицируемый**: Скрипты обеспечивают объективную проверку
* **Обратимое планирование**: Claude может итерировать план, не трогая оригиналы
* **Понятная отладка**: Сообщения об ошибках указывают на конкретные проблемы

**Когда использовать**: Пакетные операции, деструктивные изменения, сложные правила валидации, высокорисковые операции.

**Совет по реализации**: Делайте скрипты валидации подробными с конкретными сообщениями об ошибках, например "Field 'signature\_date' not found. Available fields: customer\_name, order\_total, signature\_date\_signed", чтобы помочь Claude исправить проблемы.

### Упаковка зависимостей

Навыки выполняются в среде исполнения кода с ограничениями платформы:

* **claude.ai**: Может устанавливать пакеты из npm и PyPI и загружать из репозиториев GitHub
* **Anthropic API**: Нет сетевого доступа и нет установки пакетов в рантайме

Перечислите необходимые пакеты в SKILL.md и убедитесь, что они доступны в [документации инструмента исполнения кода](/en/docs/agents-and-tools/tool-use/code-execution-tool).

### Среда выполнения

Навыки работают в среде исполнения кода с доступом к файловой системе, bash-командам и возможностями исполнения кода. Концептуальное объяснение этой архитектуры см. в [Архитектура навыков](/en/docs/agents-and-tools/agent-skills/overview#the-skills-architecture) обзора.

**Как это влияет на ваш подход:**

**Как Claude получает доступ к навыкам:**

1. **Метаданные предзагружены**: При запуске имя и описание из YAML-метаданных всех навыков загружаются в системный промпт
2. **Файлы читаются по запросу**: Claude использует bash-инструменты чтения для доступа к SKILL.md и другим файлам из файловой системы по необходимости
3. **Скрипты выполняются эффективно**: Утилитные скрипты выполняются через bash без загрузки их полного содержимого в контекст. Токены потребляет только вывод скрипта
4. **Нет штрафа за большие файлы**: Справочные файлы, данные или документация не потребляют токены контекста до фактического чтения

* **Пути к файлам важны**: Claude навигирует по каталогу навыка как по файловой системе. Используйте прямые слэши (`reference/guide.md`), не обратные
* **Давайте файлам описательные имена**: Используйте имена, указывающие на содержимое: `form_validation_rules.md`, а не `doc2.md`
* **Организуйте для обнаружения**: Структурируйте каталоги по предметным областям или функциям
  * Хорошо: `reference/finance.md`, `reference/sales.md`
  * Плохо: `docs/file1.md`, `docs/file2.md`
* **Добавляйте исчерпывающие ресурсы**: Включайте полные документы API, обширные примеры, большие наборы данных; нет штрафа контекста до обращения
* **Предпочитайте скрипты для детерминированных операций**: Напишите `validate_form.py` вместо того, чтобы просить Claude генерировать код валидации
* **Делайте намерение выполнения явным**:
  * «Запустите `analyze_form.py` для извлечения полей» (выполнение)
  * «См. `analyze_form.py` для алгоритма извлечения» (чтение как справка)
* **Тестируйте паттерны доступа к файлам**: Проверьте, что Claude может навигировать по вашей структуре каталогов, протестировав на реальных запросах

**Пример:**

```
bigquery-skill/
├── SKILL.md (обзор, ссылки на справочные файлы)
└── reference/
    ├── finance.md (метрики выручки)
    ├── sales.md (данные воронки)
    └── product.md (аналитика использования)
```

Когда пользователь спрашивает о выручке, Claude читает SKILL.md, видит ссылку на `reference/finance.md` и вызывает bash для чтения только этого файла. Файлы sales.md и product.md остаются в файловой системе, потребляя ноль токенов контекста до момента обращения. Эта модель на основе файловой системы обеспечивает постепенное раскрытие. Claude может навигировать и выборочно загружать именно то, что требуется для каждой задачи.

Полные подробности о технической архитектуре см. в [Как работают навыки](/en/docs/agents-and-tools/agent-skills/overview#how-skills-work) обзора навыков.

### Ссылки на MCP-инструменты

Если ваш навык использует MCP-инструменты (Model Context Protocol), всегда используйте полные квалифицированные имена инструментов, чтобы избежать ошибок «инструмент не найден».

**Формат**: `ИмяСервера:имя_инструмента`

**Пример**:

```markdown  theme={null}
Use the BigQuery:bigquery_schema tool to retrieve table schemas.
Use the GitHub:create_issue tool to create issues.
```

Где:

* `BigQuery` и `GitHub` — имена MCP-серверов
* `bigquery_schema` и `create_issue` — имена инструментов на этих серверах

Без префикса сервера Claude может не найти инструмент, особенно когда доступно несколько MCP-серверов.

### Не предполагайте, что инструменты установлены

Не предполагайте, что пакеты доступны:

````markdown  theme={null}
**Плохой пример: предполагает установку**:
"Используйте библиотеку pdf для обработки файла."

**Хороший пример: явно о зависимостях**:
"Установите необходимый пакет: `pip install pypdf`

Затем используйте его:
```python
from pypdf import PdfReader
reader = PdfReader("file.pdf")
```"
````

## Технические замечания

### Требования к YAML-метаданным

Метаданные SKILL.md включают только поля `name` (максимум 64 символа) и `description` (максимум 1024 символа). Полные детали структуры см. в [обзоре навыков](/en/docs/agents-and-tools/agent-skills/overview#skill-structure).

### Бюджеты токенов

Держите тело SKILL.md до 500 строк для оптимальной производительности. Если контент превышает этот лимит, разделите его на отдельные файлы, используя описанные ранее паттерны постепенного раскрытия. Подробности архитектуры см. в [обзоре навыков](/en/docs/agents-and-tools/agent-skills/overview#how-skills-work).
