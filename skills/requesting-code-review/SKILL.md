# Запрос код-ревью

Отправьте subagent superpowers:code-reviewer для выявления проблем до их накопления.

**Ключевой принцип:** Ревьюируй рано, ревьюируй часто.

## Когда запрашивать ревью

**Обязательно:**
- После каждой задачи в subagent-driven development
- После завершения крупной фичи
- Перед merge в main

**Необязательно, но полезно:**
- Когда застряли (свежий взгляд)
- Перед refactoring (базовая проверка)
- После исправления сложного бага

## Как запросить

**1. Получите git SHA:**
```bash
BASE_SHA=$(git rev-parse HEAD~1)  # или origin/main
HEAD_SHA=$(git rev-parse HEAD)
```

**2. Отправьте subagent code-reviewer:**

Используйте Task с типом superpowers:code-reviewer, заполните шаблон из `code-reviewer.md`

**Плейсхолдеры:**
- `{WHAT_WAS_IMPLEMENTED}` — что вы только что реализовали
- `{PLAN_OR_REQUIREMENTS}` — что это должно делать
- `{BASE_SHA}` — начальный commit
- `{HEAD_SHA}` — конечный commit
- `{DESCRIPTION}` — краткое описание

**3. Действуйте по обратной связи:**
- Исправляйте Critical проблемы немедленно
- Исправляйте Important проблемы перед продолжением
- Заметьте Minor проблемы на потом
- Возражайте, если ревьюер неправ (с аргументацией)

## Пример

```
[Только что завершена Задача 2: Добавить функцию верификации]

Вы: Запрошу код-ревью перед продолжением.

BASE_SHA=$(git log --oneline | grep "Task 1" | head -1 | awk '{print $1}')
HEAD_SHA=$(git rev-parse HEAD)

[Отправка subagent superpowers:code-reviewer]
  WHAT_WAS_IMPLEMENTED: Функции верификации и восстановления для индекса диалогов
  PLAN_OR_REQUIREMENTS: Задача 2 из docs/plans/deployment-plan.md
  BASE_SHA: a7981ec
  HEAD_SHA: 3df7661
  DESCRIPTION: Добавлены verifyIndex() и repairIndex() с 4 типами проблем

[Subagent возвращает]:
  Сильные стороны: Чистая архитектура, настоящие тесты
  Проблемы:
    Important: Отсутствуют индикаторы прогресса
    Minor: Магическое число (100) для интервала отчётности
  Оценка: Можно продолжать

Вы: [Исправляем индикаторы прогресса]
[Переходим к Задаче 3]
```

## Интеграция с рабочими процессами

**Subagent-Driven Development:**
- Ревью после КАЖДОЙ задачи
- Ловим проблемы до их накопления
- Исправляем перед переходом к следующей задаче

**Executing Plans:**
- Ревью после каждого пакета (3 задачи)
- Получаем обратную связь, применяем, продолжаем

**Разработка Ad-Hoc:**
- Ревью перед merge
- Ревью когда застряли

## Красные флаги

**Никогда:**
- Пропускать ревью потому что «это просто»
- Игнорировать Critical проблемы
- Продолжать с неисправленными Important проблемами
- Спорить с обоснованной технической обратной связью

**Если ревьюер неправ:**
- Возразите с технической аргументацией
- Покажите код/тесты, которые доказывают работоспособность
- Запросите уточнение
