## Чек-лист ревью

**Качество кода:**
- Чёткое разделение ответственности?
- Корректная обработка ошибок?
- Типобезопасность (если применимо)?
- Принцип DRY соблюдён?
- Граничные случаи обработаны?

**Архитектура:**
- Обоснованные проектные решения?
- Учтена масштабируемость?
- Влияние на производительность?
- Вопросы безопасности?

**Тестирование:**
- Тесты проверяют реальную логику (а не моки)?
- Покрыты граничные случаи?
- Интеграционные тесты там, где нужно?
- Все тесты проходят?

**Требования:**
- Все требования плана выполнены?
- Реализация соответствует спецификации?
- Нет расползания скоупа?
- Ломающие изменения задокументированы?

**Готовность к продакшену:**
- Стратегия миграции (если есть изменения схемы)?
- Учтена обратная совместимость?
- Документация полная?
- Нет очевидных багов?

## Критические правила

**ДЕЛАЙТЕ:**
- Категоризируйте по реальной серьёзности (не всё является критическим)
- Будьте конкретны (файл:строка, а не расплывчато)
- Объясняйте ПОЧЕМУ проблемы важны
- Отмечайте сильные стороны
- Давайте чёткий вердикт

**НЕ ДЕЛАЙТЕ:**
- Не говорите «выглядит хорошо» без проверки
- Не отмечайте мелочи как критические
- Не давайте фидбек на код, который не проверяли
- Не будьте расплывчаты («улучшить обработку ошибок»)
- Не уклоняйтесь от чёткого вердикта
