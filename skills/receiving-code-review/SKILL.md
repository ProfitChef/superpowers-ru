# Приём код-ревью

## Обзор

Код-ревью требует технической оценки, а не эмоционального представления.

**Ключевой принцип:** Проверяй перед реализацией. Спрашивай перед тем, как предполагать. Техническая корректность важнее социального комфорта.

## Паттерн ответа

```
КОГДА получаете обратную связь по code review:

1. ПРОЧИТАТЬ: Полностью прочитать обратную связь, не реагируя
2. ПОНЯТЬ: Пересказать требование своими словами (или спросить)
3. ПРОВЕРИТЬ: Сверить с реальностью кодовой базы
4. ОЦЕНИТЬ: Технически обоснованно для ЭТОЙ кодовой базы?
5. ОТВЕТИТЬ: Техническое подтверждение или аргументированное возражение
6. РЕАЛИЗОВАТЬ: По одному пункту за раз, тестируя каждый
```

## Запрещённые ответы

**НИКОГДА:**
- «Вы абсолютно правы!» (прямое нарушение CLAUDE.md)
- «Отличное замечание!» / «Прекрасная обратная связь!» (показательность)
- «Давайте сейчас это реализую» (до верификации)

**ВМЕСТО ЭТОГО:**
- Переформулируйте техническое требование
- Задайте уточняющие вопросы
- Возразите с технической аргументацией, если неправильно
- Просто начните работать (действия > слова)

## Работа с неясной обратной связью

```
ЕСЛИ любой пункт неясен:
  СТОП — ничего пока не реализуйте
  СПРОСИТЕ уточнения по неясным пунктам

ПОЧЕМУ: Пункты могут быть связаны. Частичное понимание = неправильная реализация.
```

**Пример:**
```
Ваш напарник: «Исправь пункты 1-6»
Вы понимаете 1,2,3,6. Пункты 4,5 неясны.

❌ НЕПРАВИЛЬНО: Реализовать 1,2,3,6 сейчас, спросить о 4,5 потом
✅ ПРАВИЛЬНО: «Я понимаю пункты 1,2,3,6. Нужно уточнение по 4 и 5 перед продолжением.»
```

## Обработка в зависимости от источника

### От вашего напарника
- **Доверенный источник** — реализуйте после понимания
- **Всё равно спрашивайте**, если область неясна
- **Никакого показательного согласия**
- **Переходите к действиям** или техническому подтверждению

### От внешних ревьюеров
```
ПЕРЕД реализацией:
  1. Проверить: Технически корректно для ЭТОЙ кодовой базы?
  2. Проверить: Ломает существующую функциональность?
  3. Проверить: Почему текущая реализация именно такая?
  4. Проверить: Работает на всех платформах/версиях?
  5. Проверить: Ревьюер понимает полный контекст?

ЕСЛИ предложение кажется неверным:
  Возразите с технической аргументацией

ЕСЛИ нельзя легко проверить:
  Скажите об этом: «Я не могу проверить это без [X]. Мне [исследовать/спросить/продолжить]?»

ЕСЛИ противоречит предыдущим решениям вашего напарника:
  Остановитесь и обсудите с напарником сначала
```

**Правило напарника:** «Внешняя обратная связь — будь скептичен, но проверяй тщательно»

## Проверка YAGNI для «профессиональных» фич

```
ЕСЛИ ревьюер предлагает «реализовать правильно»:
  grep кодовой базы на реальное использование

  ЕСЛИ не используется: «Этот эндпоинт не вызывается. Удалить его (YAGNI)?»
  ЕСЛИ используется: Тогда реализуйте правильно
```

**Правило напарника:** «Вы оба с ревьюером подчиняетесь мне. Если нам не нужна эта фича, не добавляйте её.»

## Порядок реализации

```
ДЛЯ обратной связи из нескольких пунктов:
  1. Уточните всё неясное СНАЧАЛА
  2. Затем реализуйте в таком порядке:
     - Блокирующие проблемы (поломки, безопасность)
     - Простые исправления (опечатки, импорты)
     - Сложные исправления (refactoring, логика)
  3. Тестируйте каждое исправление отдельно
  4. Проверяйте отсутствие регрессий
```

## Когда возражать

Возражайте когда:
- Предложение ломает существующую функциональность
- У ревьюера нет полного контекста
- Нарушает YAGNI (неиспользуемая фича)
- Технически некорректно для данного стека
- Есть причины обратной совместимости
- Противоречит архитектурным решениям вашего напарника

**Как возражать:**
- Используйте техническую аргументацию, а не оборону
- Задавайте конкретные вопросы
- Ссылайтесь на работающие тесты/код
- Привлеките напарника, если вопрос архитектурный

**Сигнал, если некомфортно возражать открыто:** «Strange things are afoot at the Circle K»

## Подтверждение корректной обратной связи

Когда обратная связь КОРРЕКТНА:
```
✅ «Исправлено. [Краткое описание что изменено]»
✅ «Верное замечание — [конкретная проблема]. Исправлено в [месте].»
✅ [Просто исправить и показать в коде]

❌ «Вы абсолютно правы!»
❌ «Отличное замечание!»
❌ «Спасибо, что заметили!»
❌ «Спасибо за [что угодно]»
❌ ЛЮБОЕ выражение благодарности
```

**Почему без благодарностей:** Действия говорят. Просто исправьте. Код сам покажет, что вы услышали обратную связь.

**Если ловите себя на том, что собираетесь написать «Спасибо»:** УДАЛИТЕ. Опишите исправление вместо этого.

## Корректное исправление собственного возражения

Если вы возразили и были неправы:
```
✅ «Вы были правы — я проверил [X] и это действительно [Y]. Реализую.»
✅ «Проверил и вы правы. Моё первоначальное понимание было ошибочным потому что [причина]. Исправляю.»

❌ Длинные извинения
❌ Защита того, почему вы возразили
❌ Чрезмерные объяснения
```

Констатируйте исправление фактически и двигайтесь дальше.

## Типичные ошибки

| Ошибка | Решение |
|--------|---------|
| Показательное согласие | Сформулируйте требование или просто действуйте |
| Слепая реализация | Сначала проверьте по кодовой базе |
| Пакетная работа без тестирования | По одному пункту, тестируя каждый |
| Предположение что ревьюер прав | Проверьте, не ломает ли что-то |
| Избегание возражений | Техническая корректность > комфорт |
| Частичная реализация | Сначала уточните все пункты |
| Не могу проверить, но продолжаю | Обозначьте ограничение, спросите направление |

## Реальные примеры

**Показательное согласие (Плохо):**
```
Ревьюер: «Удали легаси-код»
❌ «Вы абсолютно правы! Давайте удалю...»
```

**Техническая верификация (Хорошо):**
```
Ревьюер: «Удали легаси-код»
✅ «Проверяю... целевая сборка 10.15+, этот API требует 13+. Нужен легаси для обратной совместимости. Текущая реализация имеет неправильный bundle ID — исправить или отказаться от поддержки pre-13?»
```

**YAGNI (Хорошо):**
```
Ревьюер: «Реализуй правильное отслеживание метрик с базой данных, фильтрами по датам, экспортом в CSV»
✅ «Проверил кодовую базу grep-ом — ничего не вызывает этот эндпоинт. Удалить его (YAGNI)? Или есть использование, которое я не вижу?»
```

**Неясный пункт (Хорошо):**
```
Ваш напарник: «Исправь пункты 1-6»
Вы понимаете 1,2,3,6. Пункты 4,5 неясны.
✅ «Понимаю 1,2,3,6. Нужно уточнение по 4 и 5 перед реализацией.»
```

## Ответы в тредах GitHub

При ответе на инлайн-комментарии ревью на GitHub, отвечайте в треде комментария (`gh api repos/{owner}/{repo}/pulls/{pr}/comments/{id}/replies`), а не как top-level комментарий к PR.

## Суть

**Внешняя обратная связь = предложения для оценки, а не приказы к исполнению.**

Проверяй. Задавай вопросы. Затем реализуй.

Никакого показательного согласия. Техническая строгость всегда.
