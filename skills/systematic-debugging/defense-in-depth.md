# Валидация эшелонированной защитой

## Обзор

Когда вы исправляете баг, вызванный невалидными данными, добавление валидации в одном месте кажется достаточным. Но эта единственная проверка может быть обойдена через другие пути выполнения, refactoring или моки.

**Основной принцип:** Валидируйте на КАЖДОМ уровне, через который проходят данные. Сделайте баг структурно невозможным.

## Почему нужно несколько уровней

Единственная валидация: «Мы исправили баг»
Несколько уровней: «Мы сделали баг невозможным»

Разные уровни ловят разные случаи:
- Валидация на входе ловит большинство багов
- Бизнес-логика ловит граничные случаи
- Защитные проверки среды предотвращают контекстно-специфичные опасности
- Отладочное логирование помогает, когда другие уровни не справляются

## Четыре уровня

### Уровень 1: Валидация на точке входа
**Назначение:** Отклонять очевидно невалидный ввод на границе API

```typescript
function createProject(name: string, workingDirectory: string) {
  if (!workingDirectory || workingDirectory.trim() === '') {
    throw new Error('workingDirectory cannot be empty');
  }
  if (!existsSync(workingDirectory)) {
    throw new Error(`workingDirectory does not exist: ${workingDirectory}`);
  }
  if (!statSync(workingDirectory).isDirectory()) {
    throw new Error(`workingDirectory is not a directory: ${workingDirectory}`);
  }
  // ... продолжение
}
```

### Уровень 2: Валидация бизнес-логики
**Назначение:** Убедиться, что данные осмысленны для данной операции

```typescript
function initializeWorkspace(projectDir: string, sessionId: string) {
  if (!projectDir) {
    throw new Error('projectDir required for workspace initialization');
  }
  // ... продолжение
}
```

### Уровень 3: Защитные проверки среды
**Назначение:** Предотвратить опасные операции в определённых контекстах

```typescript
async function gitInit(directory: string) {
  // В тестах запрещаем git init за пределами временных директорий
  if (process.env.NODE_ENV === 'test') {
    const normalized = normalize(resolve(directory));
    const tmpDir = normalize(resolve(tmpdir()));

    if (!normalized.startsWith(tmpDir)) {
      throw new Error(
        `Refusing git init outside temp dir during tests: ${directory}`
      );
    }
  }
  // ... продолжение
}
```

### Уровень 4: Отладочная инструментация
**Назначение:** Захватывать контекст для расследования

```typescript
async function gitInit(directory: string) {
  const stack = new Error().stack;
  logger.debug('About to git init', {
    directory,
    cwd: process.cwd(),
    stack,
  });
  // ... продолжение
}
```

## Применение паттерна

Когда вы находите баг:

1. **Проследите поток данных** — откуда берётся неверное значение? Где используется?
2. **Определите все контрольные точки** — перечислите каждую точку, через которую проходят данные
3. **Добавьте валидацию на каждом уровне** — вход, бизнес-логика, среда, отладка
4. **Протестируйте каждый уровень** — попытайтесь обойти уровень 1, убедитесь, что уровень 2 ловит

## Пример из сессии

Баг: пустой `projectDir` вызывал `git init` в исходном коде

**Поток данных:**
1. Настройка теста → пустая строка
2. `Project.create(name, '')`
3. `WorkspaceManager.createWorkspace('')`
4. `git init` выполняется в `process.cwd()`

**Добавлены четыре уровня:**
- Уровень 1: `Project.create()` валидирует непустоту/существование/доступность для записи
- Уровень 2: `WorkspaceManager` валидирует, что projectDir не пуст
- Уровень 3: `WorktreeManager` запрещает git init за пределами tmpdir в тестах
- Уровень 4: Логирование стек-трейса перед git init

**Результат:** Все 1847 тестов прошли, баг невозможно воспроизвести

## Ключевой вывод

Все четыре уровня были необходимы. Во время тестирования каждый уровень ловил баги, которые пропускали остальные:
- Разные пути выполнения обходили валидацию на входе
- Моки обходили проверки бизнес-логики
- Граничные случаи на разных платформах требовали защитных проверок среды
- Отладочное логирование выявляло структурные ошибки использования

**Не останавливайтесь на одной точке валидации.** Добавляйте проверки на каждом уровне.
