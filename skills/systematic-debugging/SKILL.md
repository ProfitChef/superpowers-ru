# Систематическая отладка

## Обзор

Случайные исправления тратят время и создают новые баги. Быстрые патчи маскируют глубинные проблемы.

**Основной принцип:** ВСЕГДА находи корневую причину, прежде чем пытаться исправлять. Исправление симптомов — это провал.

**Нарушение буквы этого процесса — это нарушение духа отладки.**

## Железный закон

```
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
```

Если ты не завершил Фазу 1, ты не можешь предлагать исправления.

## Когда использовать

Используй при ЛЮБОЙ технической проблеме:
- Падения тестов
- Баги в продакшене
- Неожиданное поведение
- Проблемы производительности
- Ошибки сборки
- Проблемы интеграции

**Используй ОСОБЕННО когда:**
- Под давлением времени (в экстренных ситуациях угадывание соблазнительно)
- «Одно быстрое исправление» кажется очевидным
- Ты уже пробовал несколько исправлений
- Предыдущее исправление не сработало
- Ты не до конца понимаешь проблему

**Не пропускай когда:**
- Проблема кажется простой (у простых багов тоже есть корневые причины)
- Ты торопишься (спешка гарантирует переделку)
- Менеджер хочет исправление СЕЙЧАС (систематический подход быстрее, чем метания)

## Четыре фазы

Ты ОБЯЗАН завершить каждую фазу перед переходом к следующей.

### Фаза 1: Исследование корневой причины

**ПЕРЕД попыткой ЛЮБОГО исправления:**

1. **Внимательно прочитай сообщения об ошибках**
   - Не пропускай ошибки или предупреждения
   - Они часто содержат точное решение
   - Читай стек-трейсы полностью
   - Отмечай номера строк, пути файлов, коды ошибок

2. **Воспроизведи стабильно**
   - Можешь ли ты вызвать баг надёжно?
   - Какие точные шаги?
   - Происходит каждый раз?
   - Если не воспроизводится → собери больше данных, не угадывай

3. **Проверь недавние изменения**
   - Что изменилось, что могло вызвать это?
   - Git diff, последние commit'ы
   - Новые зависимости, изменения конфигурации
   - Различия окружения

4. **Собери улики в многокомпонентных системах**

   **КОГДА система имеет несколько компонентов (CI → сборка → подписание, API → сервис → база данных):**

   **ПЕРЕД предложением исправлений добавь диагностическую инструментацию:**
   ```
   Для КАЖДОЙ границы компонентов:
     - Логируй, какие данные входят в компонент
     - Логируй, какие данные выходят из компонента
     - Проверь распространение окружения/конфигурации
     - Проверь состояние на каждом уровне

   Запусти один раз для сбора улик, показывающих ГДЕ ломается
   ЗАТЕМ проанализируй улики для определения сбойного компонента
   ЗАТЕМ исследуй этот конкретный компонент
   ```

   **Пример (многослойная система):**
   ```bash
   # Layer 1: Workflow
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # Layer 2: Build script
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # Layer 3: Signing script
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # Layer 4: Actual signing
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **Это показывает:** Какой слой сбоит (secrets → workflow ✓, workflow → build ✗)

5. **Проследи поток данных**

   **КОГДА ошибка глубоко в стеке вызовов:**

   Смотри `root-cause-tracing.md` в этом каталоге для полной техники обратной трассировки.

   **Быстрая версия:**
   - Где возникает плохое значение?
   - Что вызвало это с плохим значением?
   - Продолжай трассировку вверх, пока не найдёшь источник
   - Исправляй в источнике, а не в симптоме

### Фаза 2: Анализ паттернов

**Найди паттерн перед исправлением:**

1. **Найди рабочие примеры**
   - Найди аналогичный работающий код в той же кодовой базе
   - Что работает из похожего на то, что сломано?

2. **Сравни с эталоном**
   - Если реализуешь паттерн, прочитай эталонную реализацию ПОЛНОСТЬЮ
   - Не просматривай бегло — читай каждую строку
   - Пойми паттерн полностью перед применением

3. **Определи различия**
   - Чем отличается рабочее от сломанного?
   - Перечисли все различия, какими бы мелкими они ни были
   - Не предполагай «это не может иметь значения»

4. **Пойми зависимости**
   - Какие ещё компоненты нужны?
   - Какие настройки, конфигурация, окружение?
   - Какие предположения оно делает?

### Фаза 3: Гипотеза и тестирование

**Научный метод:**

1. **Сформулируй одну гипотезу**
   - Сформулируй ясно: «Я думаю, X — корневая причина, потому что Y»
   - Запиши это
   - Будь конкретным, не размытым

2. **Проверь минимально**
   - Сделай НАИМЕНЬШЕЕ возможное изменение для проверки гипотезы
   - Одна переменная за раз
   - Не исправляй несколько вещей одновременно

3. **Проверь перед продолжением**
   - Сработало? Да → Фаза 4
   - Не сработало? Сформулируй НОВУЮ гипотезу
   - НЕ добавляй ещё исправлений сверху

4. **Когда не знаешь**
   - Скажи «Я не понимаю X»
   - Не притворяйся, что знаешь
   - Попроси помощи
   - Исследуй дальше

### Фаза 4: Реализация

**Исправляй корневую причину, а не симптом:**

1. **Создай падающий тест-кейс**
   - Простейшее возможное воспроизведение
   - Автоматизированный тест, если возможно
   - Одноразовый тестовый скрипт, если нет фреймворка
   - ОБЯЗАТЕЛЬНО иметь перед исправлением
   - Используй навык `superpowers:test-driven-development` для написания правильных падающих тестов

2. **Реализуй одно исправление**
   - Устрани определённую корневую причину
   - ОДНО изменение за раз
   - Никаких «раз уж я тут» улучшений
   - Никакого bundled refactoring

3. **Проверь исправление**
   - Тест теперь проходит?
   - Остальные тесты не сломались?
   - Проблема действительно решена?

4. **Если исправление не сработало**
   - СТОП
   - Посчитай: Сколько исправлений ты уже пробовал?
   - Если < 3: Вернись к Фазе 1, проанализируй заново с новой информацией
   - **Если ≥ 3: СТОП и поставь под вопрос архитектуру (шаг 5 ниже)**
   - НЕ пробуй Исправление №4 без обсуждения архитектуры

5. **Если 3+ исправления не сработали: Поставь под вопрос архитектуру**

   **Паттерн, указывающий на архитектурную проблему:**
   - Каждое исправление выявляет новое разделяемое состояние/связанность/проблему в другом месте
   - Исправления требуют «масштабного рефакторинга» для реализации
   - Каждое исправление создаёт новые симптомы в другом месте

   **СТОП и поставь под вопрос фундаментальные основы:**
   - Этот паттерн принципиально правильный?
   - Мы «продолжаем по инерции»?
   - Стоит ли рефакторить архитектуру вместо продолжения исправления симптомов?

   **Обсуди с партнёром-человеком перед дальнейшими попытками исправления**

   Это НЕ провалившаяся гипотеза — это неправильная архитектура.

## Тревожные сигналы — СТОП и следуй процессу

Если ты ловишь себя на мыслях:
- «Быстрое исправление сейчас, разберёмся потом»
- «Просто попробую изменить X и посмотрю, сработает ли»
- «Добавлю несколько изменений, прогоню тесты»
- «Пропущу тест, проверю вручную»
- «Наверное это X, давай исправлю»
- «Я не до конца понимаю, но может сработать»
- «Паттерн говорит X, но я адаптирую по-другому»
- «Вот основные проблемы: [перечисляет исправления без исследования]»
- Предлагаешь решения до трассировки потока данных
- **«Ещё одна попытка исправления» (когда уже пробовал 2+)**
- **Каждое исправление выявляет новую проблему в другом месте**

**ВСЁ это означает: СТОП. Вернись к Фазе 1.**

**Если 3+ исправления не сработали:** Поставь под вопрос архитектуру (см. Фаза 4.5)

## Сигналы от партнёра-человека, что ты делаешь неправильно

**Следи за такими перенаправлениями:**
- «Это разве не происходит?» — Ты предположил, не проверив
- «Покажет ли это нам...?» — Ты должен был добавить сбор улик
- «Хватит гадать» — Ты предлагаешь исправления без понимания
- «Подумай глубже» — Ставь под вопрос фундаментальные основы, а не только симптомы
- «Мы застряли?» (раздражённо) — Твой подход не работает

**Когда видишь такое:** СТОП. Вернись к Фазе 1.

## Типичные рационализации

| Отговорка | Реальность |
|-----------|------------|
| «Проблема простая, процесс не нужен» | У простых проблем тоже есть корневые причины. Процесс быстр для простых багов. |
| «Экстренная ситуация, нет времени на процесс» | Систематическая отладка БЫСТРЕЕ, чем метод «угадай и проверь». |
| «Сначала попробую это, потом исследую» | Первое исправление задаёт паттерн. Делай правильно с самого начала. |
| «Напишу тест после подтверждения, что исправление работает» | Непротестированные исправления не держатся. Тест-до это доказывает. |
| «Несколько исправлений сразу экономит время» | Нельзя определить, что сработало. Создаёт новые баги. |
| «Документация слишком длинная, адаптирую паттерн» | Частичное понимание гарантирует баги. Прочитай полностью. |
| «Я вижу проблему, давай исправлю» | Видеть симптомы ≠ понимать корневую причину. |
| «Ещё одна попытка» (после 2+ неудач) | 3+ неудач = архитектурная проблема. Ставь под вопрос паттерн, а не пытайся снова. |

## Краткая справка

| Фаза | Ключевые действия | Критерии успеха |
|------|-------------------|-----------------|
| **1. Корневая причина** | Читай ошибки, воспроизводи, проверяй изменения, собирай улики | Понять ЧТО и ПОЧЕМУ |
| **2. Паттерн** | Найди рабочие примеры, сравни | Определить различия |
| **3. Гипотеза** | Сформулируй теорию, проверь минимально | Подтверждена или новая гипотеза |
| **4. Реализация** | Создай тест, исправь, проверь | Баг устранён, тесты проходят |

## Когда процесс показывает «нет корневой причины»

Если систематическое исследование показывает, что проблема действительно связана с окружением, зависит от тайминга или является внешней:

1. Ты завершил процесс
2. Задокументируй, что ты исследовал
3. Реализуй соответствующую обработку (повтор, таймаут, сообщение об ошибке)
4. Добавь мониторинг/логирование для дальнейшего исследования

**Но:** 95% случаев «нет корневой причины» — это незавершённое исследование.

## Влияние в реальной жизни

Из сессий отладки:
- Систематический подход: 15-30 минут до исправления
- Подход случайных исправлений: 2-3 часа метаний
- Частота исправления с первого раза: 95% против 40%
- Внесённые новые баги: почти ноль против частого явления
