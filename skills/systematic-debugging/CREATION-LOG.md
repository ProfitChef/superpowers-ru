# Журнал создания: навык систематической отладки

Справочный пример извлечения, структурирования и защиты критического навыка от обхода.

## Исходный материал

Фреймворк отладки извлечён из `/Users/jesse/.claude/CLAUDE.md`:
- 4-фазный систематический процесс (Расследование → Анализ паттернов → Гипотеза → Реализация)
- Основной мандат: ВСЕГДА находить корневую причину, НИКОГДА не исправлять симптомы
- Правила, разработанные для сопротивления давлению времени и рационализации

## Решения при извлечении

**Что включить:**
- Полный 4-фазный фреймворк со всеми правилами
- Анти-шорткаты («НИКОГДА не исправляй симптом», «ОСТАНОВИСЬ и пересмотри анализ»)
- Язык, устойчивый к давлению («даже если быстрее», «даже если я тороплюсь»)
- Конкретные шаги для каждой фазы

**Что не включать:**
- Контекст, специфичный для проекта
- Повторяющиеся вариации одного правила
- Описательные объяснения (сжаты до принципов)

## Структура по skill-creation/SKILL.md

1. **Развёрнутый when_to_use** — включены симптомы и анти-паттерны
2. **Тип: техника** — конкретный процесс с шагами
3. **Ключевые слова** — «корневая причина», «симптом», «обходное решение», «отладка», «расследование»
4. **Блок-схема** — точка принятия решения «фикс не сработал» → пересмотреть анализ vs добавить ещё фиксов
5. **Пофазная разбивка** — формат сканируемого чек-листа
6. **Раздел анти-паттернов** — чего НЕ делать (критически важно для этого навыка)

## Элементы защиты от обхода

Фреймворк разработан для сопротивления рационализации под давлением:

### Выбор формулировок
- «ВСЕГДА» / «НИКОГДА» (не «следует» / «старайтесь»)
- «даже если быстрее» / «даже если я тороплюсь»
- «ОСТАНОВИСЬ и пересмотри анализ» (явная пауза)
- «Не проскакивай мимо» (ловит реальное поведение)

### Структурная защита
- **Фаза 1 обязательна** — нельзя перескочить к реализации
- **Правило одной гипотезы** — заставляет думать, предотвращает фиксы «дробовиком»
- **Явный сценарий неудачи** — «ЕСЛИ первый фикс не сработал» с обязательным действием
- **Раздел анти-паттернов** — показывает, как именно выглядят шорткаты

### Избыточность
- Мандат корневой причины в обзоре + when_to_use + Фаза 1 + правила реализации
- «НИКОГДА не исправляй симптом» появляется 4 раза в разных контекстах
- У каждой фазы есть явное указание «не пропускай»

## Подход к тестированию

Создано 4 валидационных теста по skills/meta/testing-skills-with-subagents:

### Тест 1: Академический контекст (без давления)
- Простой баг, нет давления по времени
- **Результат:** Идеальное соответствие, полное расследование

### Тест 2: Давление времени + очевидный быстрый фикс
- Пользователь «торопится», исправление симптома выглядит простым
- **Результат:** Устоял перед шорткатом, прошёл полный процесс, нашёл реальную корневую причину

### Тест 3: Сложная система + неопределённость
- Многоуровневый сбой, неясно, удастся ли найти корневую причину
- **Результат:** Систематическое расследование, проследил все уровни, нашёл источник

### Тест 4: Неудачный первый фикс
- Гипотеза не сработала, соблазн добавить ещё фиксов
- **Результат:** Остановился, пересмотрел анализ, сформулировал новую гипотезу (без «дробовика»)

**Все тесты пройдены.** Рационализаций не обнаружено.

## Ключевой вывод

**Самый важный элемент защиты:** раздел анти-паттернов, показывающий конкретные шорткаты, которые кажутся оправданными в момент принятия решения. Когда Claude думает «Я просто добавлю этот один быстрый фикс», увидеть именно этот паттерн в списке ошибочных создаёт когнитивное трение.

## Пример использования

При обнаружении бага:
1. Загрузить навык: skills/debugging/systematic-debugging
2. Прочитать обзор (10 сек) — напоминание о мандате
3. Следовать чек-листу Фазы 1 — принудительное расследование
4. Если возникает соблазн пропустить — увидеть анти-паттерн, остановиться
5. Пройти все фазы — корневая причина найдена

**Инвестиция времени:** 5-10 минут
**Сэкономленное время:** часы игры в «ударь крота» с симптомами
