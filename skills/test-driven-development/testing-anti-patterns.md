# Антипаттерны тестирования

**Загружайте этот справочник когда:** пишете или изменяете тесты, добавляете моки, или испытываете соблазн добавить тестовые методы в продакшен-код.

## Обзор

Тесты должны проверять реальное поведение, а не поведение моков. Моки — это средство изоляции, а не то, что тестируется.

**Основной принцип:** Тестируйте то, что делает код, а не то, что делают моки.

**Строгое следование TDD предотвращает эти антипаттерны.**

## Железные законы

```
1. НИКОГДА не тестируйте поведение моков
2. НИКОГДА не добавляйте тестовые методы в продакшен-классы
3. НИКОГДА не мокайте без понимания зависимостей
```

## Антипаттерн 1: Тестирование поведения моков

**Нарушение:**
```typescript
// ❌ ПЛОХО: Тестируем, что мок существует
test('renders sidebar', () => {
  render(<Page />);
  expect(screen.getByTestId('sidebar-mock')).toBeInTheDocument();
});
```

**Почему это неправильно:**
- Вы проверяете работу мока, а не компонента
- Тест проходит, когда мок есть, падает, когда его нет
- Ничего не говорит о реальном поведении

**Замечание вашего напарника:** «Мы тестируем поведение мока?»

**Исправление:**
```typescript
// ✅ ХОРОШО: Тестируйте реальный компонент или не мокайте его
test('renders sidebar', () => {
  render(<Page />);  // Не мокайте sidebar
  expect(screen.getByRole('navigation')).toBeInTheDocument();
});

// ИЛИ если sidebar нужно замокать для изоляции:
// Не проверяйте мок — тестируйте поведение Page с sidebar
```

### Контрольная функция

```
ПЕРЕД проверкой любого мок-элемента:
  Спросите: «Я тестирую реальное поведение компонента или просто наличие мока?»

  ЕСЛИ наличие мока:
    СТОП — Удалите проверку или уберите мок компонента

  Тестируйте реальное поведение
```

## Антипаттерн 2: Тестовые методы в продакшен-коде

**Нарушение:**
```typescript
// ❌ ПЛОХО: destroy() используется только в тестах
class Session {
  async destroy() {  // Выглядит как продакшен-API!
    await this._workspaceManager?.destroyWorkspace(this.id);
    // ... очистка
  }
}

// В тестах
afterEach(() => session.destroy());
```

**Почему это неправильно:**
- Продакшен-класс загрязнён тестовым кодом
- Опасно при случайном вызове в продакшене
- Нарушает YAGNI и разделение ответственности
- Путает жизненный цикл объекта с жизненным циклом сущности

**Исправление:**
```typescript
// ✅ ХОРОШО: Тестовые утилиты занимаются очисткой тестов
// У Session нет destroy() — в продакшене он stateless

// В test-utils/
export async function cleanupSession(session: Session) {
  const workspace = session.getWorkspaceInfo();
  if (workspace) {
    await workspaceManager.destroyWorkspace(workspace.id);
  }
}

// В тестах
afterEach(() => cleanupSession(session));
```

### Контрольная функция

```
ПЕРЕД добавлением любого метода в продакшен-класс:
  Спросите: «Этот метод используется только тестами?»

  ЕСЛИ да:
    СТОП — Не добавляйте его
    Поместите в тестовые утилиты

  Спросите: «Этот класс владеет жизненным циклом этого ресурса?»

  ЕСЛИ нет:
    СТОП — Не тот класс для этого метода
```

## Антипаттерн 3: Мокание без понимания

**Нарушение:**
```typescript
// ❌ ПЛОХО: Мок ломает логику теста
test('detects duplicate server', () => {
  // Мок предотвращает запись конфига, от которой зависит тест!
  vi.mock('ToolCatalog', () => ({
    discoverAndCacheTools: vi.fn().mockResolvedValue(undefined)
  }));

  await addServer(config);
  await addServer(config);  // Должно выбросить ошибку — но не выбросит!
});
```

**Почему это неправильно:**
- Замоканный метод имел побочный эффект, от которого зависел тест (запись конфига)
- Избыточное мокание «на всякий случай» ломает реальное поведение
- Тест проходит по неправильной причине или загадочно падает

**Исправление:**
```typescript
// ✅ ХОРОШО: Мокайте на правильном уровне
test('detects duplicate server', () => {
  // Мокаем медленную часть, сохраняем поведение, нужное тесту
  vi.mock('MCPServerManager'); // Мокаем только медленный запуск сервера

  await addServer(config);  // Конфиг записан
  await addServer(config);  // Дубликат обнаружен ✓
});
```

### Контрольная функция

```
ПЕРЕД моканием любого метода:
  СТОП — Не мокайте пока

  1. Спросите: «Какие побочные эффекты у реального метода?»
  2. Спросите: «Зависит ли этот тест от каких-либо из этих побочных эффектов?»
  3. Спросите: «Я полностью понимаю, что нужно этому тесту?»

  ЕСЛИ зависит от побочных эффектов:
    Мокайте на более низком уровне (саму медленную/внешнюю операцию)
    ИЛИ используйте тестовые двойники, сохраняющие нужное поведение
    НЕ высокоуровневый метод, от которого зависит тест

  ЕСЛИ не уверены, что нужно тесту:
    Запустите тест с реальной реализацией СНАЧАЛА
    Посмотрите, что реально должно произойти
    ПОТОМ добавьте минимальное мокание на правильном уровне

  Тревожные сигналы:
    - «Замокаю на всякий случай»
    - «Может быть медленно, лучше замокать»
    - Мокание без понимания цепочки зависимостей
```

## Антипаттерн 4: Неполные моки

**Нарушение:**
```typescript
// ❌ ПЛОХО: Частичный мок — только поля, которые вы считаете нужными
const mockResponse = {
  status: 'success',
  data: { userId: '123', name: 'Alice' }
  // Отсутствует: metadata, используемая нижестоящим кодом
};

// Позже: ломается, когда код обращается к response.metadata.requestId
```

**Почему это неправильно:**
- **Частичные моки скрывают структурные допущения** — вы замокали только известные вам поля
- **Нижестоящий код может зависеть от полей, которые вы не включили** — тихие ошибки
- **Тесты проходят, но интеграция ломается** — мок неполный, реальный API полный
- **Ложная уверенность** — тест ничего не доказывает о реальном поведении

**Железное правило:** Мокайте ПОЛНУЮ структуру данных, как она существует в реальности, а не только поля, которые использует ваш непосредственный тест.

**Исправление:**
```typescript
// ✅ ХОРОШО: Воспроизводите полноту реального API
const mockResponse = {
  status: 'success',
  data: { userId: '123', name: 'Alice' },
  metadata: { requestId: 'req-789', timestamp: 1234567890 }
  // Все поля, которые возвращает реальный API
};
```

### Контрольная функция

```
ПЕРЕД созданием мок-ответов:
  Проверьте: «Какие поля содержит реальный ответ API?»

  Действия:
    1. Изучите реальный ответ API из документации/примеров
    2. Включите ВСЕ поля, которые система может использовать ниже по цепочке
    3. Убедитесь, что мок полностью соответствует схеме реального ответа

  Критично:
    Если вы создаёте мок, вы должны понимать ВСЮУ структуру
    Частичные моки тихо ломаются, когда код зависит от пропущенных полей

  Если не уверены: Включите все задокументированные поля
```

## Антипаттерн 5: Интеграционные тесты как запоздалая мысль

**Нарушение:**
```
✅ Реализация завершена
❌ Тесты не написаны
«Готово к тестированию»
```

**Почему это неправильно:**
- Тестирование — часть реализации, а не необязательное продолжение
- TDD поймал бы это
- Нельзя заявлять о завершении без тестов

**Исправление:**
```
Цикл TDD:
1. Напишите падающий тест
2. Реализуйте до прохождения
3. Проведите refactoring
4. ТОГДА заявляйте о завершении
```

## Когда моки становятся слишком сложными

**Предупреждающие сигналы:**
- Настройка моков длиннее логики теста
- Мокание всего подряд, чтобы тест прошёл
- У моков отсутствуют методы реальных компонентов
- Тест ломается при изменении мока

**Вопрос вашего напарника:** «Нам вообще нужен здесь мок?»

**Подумайте:** Интеграционные тесты с реальными компонентами часто проще сложных моков

## TDD предотвращает эти антипаттерны

**Почему TDD помогает:**
1. **Сначала напишите тест** → Заставляет задуматься, что именно вы тестируете
2. **Посмотрите, как он падает** → Подтверждает, что тест проверяет реальное поведение, а не моки
3. **Минимальная реализация** → Тестовые методы не просачиваются в код
4. **Реальные зависимости** → Вы видите, что тесту реально нужно, до мокания

**Если вы тестируете поведение мока — вы нарушили TDD** — вы добавили моки, не посмотрев сначала, как тест падает на реальном коде.

## Краткий справочник

| Антипаттерн | Исправление |
|-------------|-------------|
| Проверки мок-элементов | Тестируйте реальный компонент или уберите мок |
| Тестовые методы в продакшене | Перенесите в тестовые утилиты |
| Мокание без понимания | Сначала поймите зависимости, мокайте минимально |
| Неполные моки | Воспроизводите реальный API полностью |
| Тесты как запоздалая мысль | TDD — сначала тесты |
| Чрезмерно сложные моки | Рассмотрите интеграционные тесты |

## Тревожные сигналы

- Проверки ищут test ID вида `*-mock`
- Методы вызываются только в тестовых файлах
- Настройка моков — более 50% теста
- Тест падает при удалении мока
- Не можете объяснить, зачем мок нужен
- Мокание «на всякий случай»

## Итог

**Моки — инструмент изоляции, а не предмет тестирования.**

Если TDD показывает, что вы тестируете поведение мока — вы пошли не туда.

Исправление: тестируйте реальное поведение или задайте вопрос, зачем вы вообще мокаете.
